# 风扇自动控制配置持久化设计

## 背景

当前应用将风扇配置数组编码到 `UserDefaults` 的 `fanProfiles` 中，但启动时只恢复数组，没有恢复 `activeProfile` 和 `isAutoControlEnabled`。自定义曲线又通过 `enableAutoControl(profile:)` 间接触发保存；当 Helper 不可用时，该方法提前返回，导致自定义曲线没有持久化。

## 目标

- 保存完整的风扇控制配置，包括配置列表、当前选中配置和自动控制启用状态。
- 应用重启后恢复上次选中的配置，并直接重新启用自动控制。
- 自定义曲线的保存不依赖 Helper 是否可用。
- 兼容已经写入 `fanProfiles` 的旧版数据。
- 保持风扇控制失败与配置持久化失败相互独立。

## 数据模型

在 `MacFanControlCore` 中新增可编码的 `FanControlSettings`：

- `profiles: [FanProfile]`
- `activeProfileID: UUID?`
- `isAutoControlEnabled: Bool`

状态整体编码后写入新的 `UserDefaults` 键。整体状态保证配置内容、选中项和启用状态来自同一次保存，不会因多个键部分更新而失配。

## 持久化边界

新增一个纯逻辑的状态编解码和恢复入口，使其可以在 `MacFanControlCoreTests` 中直接测试。`FanController` 负责：

1. 启动时读取新格式状态。
2. 如果没有新格式，读取旧 `fanProfiles`，迁移成新格式并写回。
3. 恢复 `profiles`、`activeProfile` 和 `isAutoControlEnabled`。
4. 在控制能力可用时启动自动控制定时器；暂不可用时保留恢复后的配置状态和选中项，后续 Helper 安装成功后自动启动。

## 自定义曲线保存流程

曲线编辑器不再直接修改数组后依赖 `enableAutoControl(profile:)` 保存。它调用 `FanController` 的专用入口：

1. 对控制点按温度排序。
2. 更新现有自定义配置，尽量保留其稳定 ID；不存在时创建。
3. 将自定义配置设为当前选中项并标记自动控制启用。
4. 立即保存完整状态。
5. Helper 可用时启动控制；不可用时报告控制不可用，但不回滚已保存配置。

稳定 ID 可以避免每次编辑自定义曲线后产生新的选中标识，也让重启恢复更可靠。

## 启动恢复

恢复状态时，以 `activeProfileID` 为唯一选中来源，并同步每个配置的 `isActive` 显示字段：

- 找到配置且自动控制已启用：恢复选中项，并准备启动自动控制。
- 找不到对应配置：清除无效选中项和启用状态，保存修正后的状态。
- 自动控制关闭：保留配置列表，但不恢复活动配置。

监控启动后，如果状态要求自动控制且 Helper 可用，则创建自动控制定时器并立即应用一次曲线。定时器创建保持幂等，避免重复启动。

## 错误处理

- 解码失败时保留默认预设，并设置 `settingsLoadFailed`。
- 编码或写入失败时设置 `settingsSaveFailed`。
- Helper 不可用只影响实际风扇控制，不影响配置保存。
- 旧格式迁移失败时不删除旧数据，避免用户配置丢失。

## 测试

新增回归测试覆盖：

- 完整状态编码和解码后，自定义曲线保持一致。
- 选中配置 ID 和自动控制启用状态能够恢复。
- 自定义配置更新时保持稳定 ID。
- 无效活动配置 ID 会被规范化为关闭状态。
- 旧 `fanProfiles` 数据可以迁移为新状态。

现有 24 个模型测试必须继续通过，并运行完整 `swift test` 验证构建和回归测试。

## 非目标

- 本次不统一 C/Socket 与 Swift/XPC 两套 Helper 架构。
- 不调整风扇曲线编辑器布局。
- 不改变预设曲线内容或温度插值算法。
